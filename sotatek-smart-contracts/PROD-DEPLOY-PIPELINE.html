2단계 · 권한 이양 — deployer EOA가 실행, 종착지는 Safe
5
① SettingManagement admin 이전 (개시)
deployer EOA
OPERATOR_SAFE_ADDRESS=0x<safe> \ npx hardhat run scripts/grant-admin-to-safe.js --network mainnet
transferAdmin(safe) — pendingAdmin만 설정(되돌릴 수 있음). 실제 전환은 Safe가 수락해야 완료.
✋
수동Safe UI — acceptAdmin()

app.safe.global (eth:<safe>) → Contract interaction → SettingManagement.acceptAdmin() → 임계값 오너 서명 → 실행. 이 순간 Safe가 SM admin이 되고, SM.hasRole로 admin을 조회하는 모든 컨트랙트(Earn 볼트×2 · 어댑터 · ReservePool · CircuitBreaker)를 자동 승계. (파이프라인 중 유일한 수동 단계)

6
② ProxyAdmin(업그레이드 권한) 이전
deployer EOA
OPERATOR_SAFE_ADDRESS=0x<safe> \ # 두 번째 볼트 프록시가 레지스트리에 없으면: PROXY_ADDRESSES=0x<vault2> \ npx hardhat run scripts/transfer-proxy-admin-to-safe.js --network mainnet
업그레이더블 프록시(SettingManagement · Earn볼트×2 · CircuitBreaker)의 ProxyAdmin 소유권 → Safe. 이게 빠지면 임시 EOA가 프록시 업그레이드 권한을 계속 보유 → 멀티시그 무력화.
7
③ Lock Registry owner 이전
deployer EOA
OPERATOR_SAFE_ADDRESS=0x<safe> \ npx hardhat run scripts/transfer-lock-registry-owner.js --network mainnet
EarnLockRegistry.setOwner(safe). relayer의 locker 권한은 유지(자금경로 비차단) — owner만 이동.

무권한 검증 → 임시 EOA 폐기
deployer EOA / 콘솔
deployer가 더 이상 (a) SettingManagement admin · (b) 어느 ProxyAdmin의 owner · (c) EarnLockRegistry owner 가 아님을 온체인 확인 → 그제서야 임시 EOA 폐기/콜드보관.
settingManagement.owner()/admin == safegetAdminAddress(proxy).owner() == safelockRegistry.owner() == safe
✅ 완료 후: Safe(멀티시그)가 모든 런칭 컨트랙트의 admin·업그레이드·owner를 단독 통제. 이후 운영 설정(keeper/pauser/쿨다운 등)은 어드민에서 누르면 Safe Queue 제안 → 오너 승인 → 실행으로 처리.
